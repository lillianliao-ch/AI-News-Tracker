# Cursor Subagent 使用指南 - AI 猎头助手项目

## 什么是 Cursor Subagent？

**Subagent** 是 Cursor 中的一个强大功能，让 AI 助手可以：

1. ✅ **长期记忆**：跨会话记住项目上下文、架构设计、开发决策
2. ✅ **模块化开发**：分模块逐步实现，而不是一次性完成
3. ✅ **任务追踪**：自动管理 TODO 列表，跟踪开发进度
4. ✅ **持续迭代**：在多次会话中持续完善代码
5. ✅ **智能建议**：基于项目历史提供针对性建议

## 🚀 Subagent 的高级用法

### 1. **跨会话持续开发**

**场景**：今天实现了插件，明天继续开发后台

**第一天**：
```
你: "帮我创建 Chrome 插件的基础框架"
Subagent: [创建代码，实现功能]
你: "很好，今天就到这里，明天继续"
```

**第二天**（新的会话）：
```
你: "继续昨天的项目，现在开始开发后台 API"
Subagent: "好的，昨天我们完成了 Chrome 插件基础框架。
          现在开始创建 FastAPI 后台项目..."
         [自动记住昨天的上下文]
```

**关键**：Subagent 会通过以下方式记住上下文：
- 项目文件结构
- PRD 文档
- 已实现的代码
- 技术选型决策

---

### 2. **智能 Debug**

**场景**：某个功能不工作

```
你: "插件的简历下载功能报错了，帮我 debug"

Subagent 会：
1. 查看相关代码文件
2. 检查错误日志
3. 分析可能原因
4. 提供修复方案
5. 更新代码
```

因为 Subagent 记住了整个项目的架构，它能快速定位问题。

---

### 3. **功能迭代**

**场景**：需要优化现有功能

```

    我想加入候选人的职业稳定性评分"

Subagent 会：
1. 找到 ai_evaluator.py

4. 更新 Prompt
5. 调整权重
6. 更新数据库字段
7. 更新飞书表结构
```

**关键**：Subagent 知道要改哪些文件，因为它记住了整个架构。

## 📌 Subagent 最佳实践

### ✅ DO（推荐做法）

1. **分阶段开发**
   ```
   ❌ "帮我实现整个项目"
   ✅ "先实现 Chrome 插件的候选人列表采集功能"
   ```

2. **明确上下文**
   ```
   ❌ "这个不工作"
   ✅ "在 content.js 的第 45 行，简历下载功能报错了"
   ```

3. **保存关键决策**
   ```
   你: "我决定使用 GPT-4o 而不是 Claude，
       因为它的 JSON 输出更稳定"
   Subagent: "好的，我已记住。后续所有 AI 集成都使用 OpenAI。"
   ```

4. **及时反馈**
   ```
   你: "这个 Prompt 生成的话术太正式了，
       我需要更轻松的语气"
   Subagent: [调整 Prompt]
   ```

### ❌ DON'T（不推荐做法）

1. **避免一次性要求太多**
   ```
   ❌ "帮我实现插件+后台+AI+飞书，一次性搞定"
   → 代码质量差，难以调试
   ```

2. **避免频繁切换上下文**
   ```
   ❌ 插件写到一半，突然说"帮我改一下数据库设计"
   → 容易混乱
   ```

3. **避免跳过基础设施**
   ```
   ❌ 直接说"帮我实现 AI 评估"，但数据库和 API 都还没搭建
   ```

## 💡 高级技巧

### 1. **让 Subagent 成为你的技术顾问**

```
你: "我在纠结是用 GPT-4o 还是 Claude 3.5 Sonnet。
    能帮我分析一下吗？"

Subagent: 
"基于你的需求（简历解析 + 匹配度评估），我推荐：

GPT-4o 优势：
✅ JSON 模式更稳定
✅ 成本更低（$0.005/1K tokens）
✅ 速度更快

Claude 3.5 Sonnet 优势：
✅ 长文本理解更好
✅ 推理能力更强
✅ 更少幻觉

推荐方案：
- 简历解析用 GPT-4o（需要稳定 JSON）
- 匹配度评估用 Claude（需要深度分析）

这样可以平衡成本和效果。"
```

### 2. **让 Subagent 帮你做技术选型**

```
你: "PDF 解析有哪些库可以选？帮我对比一下"

Subagent: [提供详细对比，并推荐方案]
```

### 3. **让 Subagent 帮你设计架构**

```
你: "我担心高并发时后台性能不够，
    应该加入消息队列吗？"

Subagent: 
"根据你的需求（每天 200 份简历），
目前不需要消息队列。

原因：
- 每天 200 份 = 每小时 ~20 份（假设8小时工作）
- 单个处理耗时 ~30 秒
- 同步处理完全够用

建议：
- 第一版用同步处理
- 如果后续扩展到 1000+ 份/天，再加入 Celery + Redis

过早优化是万恶之源。先做出来，再优化。"
```
