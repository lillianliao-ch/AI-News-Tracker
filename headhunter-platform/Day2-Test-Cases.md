## 1. 用户注册功能测试

### 1.1 正常注册流程测试

#### Test Case 1.1.1: 顾问用户注册
**步骤:**
1. 访问 http://localhost:3000/register
2. 填写表单：
   - 用户名: consultant_test
   - 邮箱: consultant@test.com
   - 手机: 13800138001
   - 角色: 顾问
   - 密码: password123
   - 确认密码: password123
3. 点击"注册"按钮

**预期结果:**
- ✅ 显示注册成功消息
- ✅ 返回用户ID
- ✅ 用户状态为"待审核"(pending)
- ✅ 表单清空

#### Test Case 1.1.2: SOHO用户注册
**步骤:**
1. 访问注册页面
2. 填写SOHO用户信息
3. 提交注册

**预期结果:**
- ✅ 注册成功
- ✅ 角色正确设置为SOHO

#### Test Case 1.1.3: 公司管理员注册
**步骤:**
1. 访问注册页面
2. 选择"公司管理员"角色
3. 填写额外的公司信息：
   - 公司名称: 测试科技有限公司
   - 营业执照号: 91110000123456789X
4. 提交注册

**预期结果:**
- ✅ 注册成功
- ✅ 公司信息同时创建
- ✅ 公司状态为"待审核"

### 1.2 注册验证测试

#### Test Case 1.2.1: 密码不匹配
**步骤:**
1. 填写表单，确认密码与密码不一致
2. 提交表单

**预期结果:**
- ❌ 显示错误消息："密码和确认密码不匹配"
- ❌ 不提交到后端

#### Test Case 1.2.2: 邮箱格式验证
**步骤:**
1. 输入无效邮箱格式 (如: "invalid-email")
2. 提交表单

**预期结果:**
- ❌ HTML5验证阻止提交
- ❌ 显示邮箱格式错误

#### Test Case 1.2.3: 公司管理员缺少公司信息
**步骤:**
1. 选择"公司管理员"角色
2. 不填写公司名称或营业执照号
3. 提交表单

**预期结果:**
- ❌ 显示错误："公司管理员角色需要填写公司名称和营业执照"

#### Test Case 1.2.4: 重复邮箱/手机
**步骤:**
1. 使用已存在的邮箱或手机号注册
2. 提交表单

**预期结果:**
- ❌ 后端返回409冲突错误
- ❌ 前端显示："User with this email or phone already exists"

## 5. API端点测试

### 5.1 认证API测试

#### Test Case 5.1.1: POST /api/auth/register
```bash
curl -X POST http://localhost:4000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "username": "test_api",
    "email": "test_api@example.com",
    "phone": "13900139001",
    "password": "password123",
    "role": "consultant"
  }'
```
**预期结果:**
- ✅ 返回201状态码
- ✅ 返回用户ID和成功消息

#### Test Case 5.1.2: POST /api/auth/login
```bash
curl -X POST http://localhost:4000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{
    "email": "admin@test.com",
    "password": "password123"
  }'
```
**预期结果:**
- ✅ 返回200状态码
- ✅ 返回JWT token和用户信息

#### Test Case 5.1.3: GET /api/auth/profile
```bash
curl -H "Authorization: Bearer [TOKEN]" \
  http://localhost:4000/api/auth/profile
```
**预期结果:**
- ✅ 返回当前用户完整信息
- ✅ 包含公司信息（如有）

### 5.2 用户管理API测试

#### Test Case 5.2.1: GET /api/auth/pending-users
**预期结果:**
- ✅ 只允许平台管理员访问
- ✅ 返回待审核用户列表
- ❌ 非管理员返回403

#### Test Case 5.2.2: PATCH /api/auth/approve-user/:userId
**预期结果:**
- ✅ 成功更新用户状态
