# 猎头协作平台技术架构设计文档

## 一、技术栈选型

### 1.1 前端技术栈

| 技术 | 选型 | 版本 | 说明 |
|------|------|------|------|
| 框架 | Next.js | 15.5.2 | React全栈框架，支持SSR/SSG，启用Turbopack |
| UI框架 | React | 19.1.0 | 最新React版本，支持Concurrent Features |
| UI样式 | Tailwind CSS | 4.0 | 原子化CSS框架，支持PostCSS插件 |
| 图标库 | Lucide React | 0.544.0 | 轻量级SVG图标库 |
| 类型系统 | TypeScript | 5.0+ | 静态类型检查 |
| 构建工具 | Turbopack (Next.js内置) | 15.5.2 | 更快的Webpack替代方案 |
| 代码检查 | ESLint | 9.0+ | 代码质量检查工具 |
| 状态管理 | React内置状态 + Context API | - | 基于组件状态管理，无需额外状态库 |

**技术选型决策 (ADR-001)：**
- **决策**：选择Next.js 15.5.2 + React 19.1.0 + Tailwind CSS 4.0
- **理由**：
  - Next.js提供完整的全栈开发体验，内置Turbopack提升构建性能
  - React 19支持最新的Concurrent Features，提升用户体验
  - Tailwind CSS 4.0原子化设计，快速开发且体积优化
  - 避免引入重量级UI库（如Ant Design），减少bundle体积
- **权衡**：放弃现成组件库，需要自行开发基础组件
- **状态**：已实施

| 技术 | 选型 | 版本 | 说明 |
|------|------|------|------|
| 运行环境 | Node.js | 18+ | 高性能JavaScript运行时 |
| Web框架 | Fastify | 4.24.3 | 高性能Web框架，内置JSON Schema验证 |
| 类型系统 | TypeScript | 5.2.2 | 静态类型检查 |
| 数据库ORM | Prisma | 5.6.0 | 类型安全的数据库客户端 |

| API文档 | @fastify/swagger + @fastify/swagger-ui | 8.12.0 + 2.1.0 | 自动化API文档生成 |
| 跨域处理 | @fastify/cors | 8.4.0 | CORS配置 |
| 静态文件 | @fastify/static | 6.12.0 | 静态文件服务 |

**技术选型决策 (ADR-002)：**
- **决策**：选择Fastify + Prisma + TypeScript技术栈
- **理由**：
  - Fastify性能优于Express，内置JSON Schema验证和Swagger文档
  - Prisma提供类型安全的数据库访问，简化SQL操作
  - TypeScript在前后端统一类型定义，提升开发效率
  - Socket.io支持实时通信需求（PM权限、消息推送）
- **权衡**：学习成本相对较高，但长期维护性更好
- **状态**：已实施

**技术选型决策 (ADR-003)：**
- **决策**：选择PostgreSQL + Redis + Prisma数据层架构
- **理由**：
  - PostgreSQL支持复杂业务逻辑和事务处理，适合B2B协作场景
  - Redis作为缓存层和Session存储，提升查询性能
  - Prisma简化数据库迁移和类型安全访问
  - Zod提供运行时数据验证，与TypeScript类型系统互补
- **权衡**：相比NoSQL方案，关系型数据库在扩展性上有限制
- **状态**：已实施

// 自适应参数选择算法
const getOptimalParameters = (imageMetrics: ImageMetrics): ProcessingOptions => {
  const options: ProcessingOptions = {
    enhance: true,
    denoise: true,
    sharpen: true,
    contrast: 1.2,
    brightness: 1.1,
    threshold: false,
    dpi: 300
  };

  // 小图片强化策略
  if (imageMetrics.width < 800) {
    options.contrast = 1.4;
    options.dpi = 400;
    options.sharpen = true;
  }

  // 暗图片增亮策略
  if (imageMetrics.meanBrightness < 100) {
    options.brightness = 1.3;
  }

  // 低对比度二值化策略
  if (imageMetrics.contrastRange < 100) {
    options.threshold = true;
  }

  return options;
};

## 十二、架构决策记录 (ADR)

### ADR-004：OCR技术选型
- **决策**：采用Tesseract.js + 云端OCR多引擎架构
- **背景**：需要处理大量职位描述图片的OCR识别
- **选项**：
  1. 纯云端OCR服务（成本高，依赖网络）
  2. 纯离线OCR（准确率低）
  3. 混合架构（推荐）
- **决策理由**：
  - Tesseract.js作为基础引擎，成本可控
  - 预留云端OCR接口，提升识别准确率
  - Sharp进行图像预处理，优化识别效果
- **状态**：已实施

### ADR-005：实时通信架构
- **决策**：采用Socket.io实现实时通信
- **背景**：PM权限管理和消息推送需要实时性
- **选项**：
  1. WebSocket原生实现
  2. Socket.io（推荐）
  3. Server-Sent Events
- **决策理由**：
  - Socket.io提供自动重连和降级机制
  - 支持房间管理，适合多公司隔离需求
  - 生态成熟，调试工具完善
- **状态**：已实施

### ADR-006：前端状态管理
- **决策**：使用React内置状态管理，避免引入额外状态库
- **背景**：项目规模适中，状态复杂度可控
- **选项**：
  1. Zustand（轻量级）
  2. Redux Toolkit（功能全面）
  3. React内置状态 + Context API（推荐）
- **决策理由**：
  - React 19的useOptimistic和useTransition满足大多数需求
  - Context API适合全局状态（用户信息、权限）
  - 减少bundle体积和学习成本
- **状态**：已实施

### 13.1 主要技术风险

| 风险类型 | 具体风险 | 影响级别 | 应对方案 |
